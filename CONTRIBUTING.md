# Contributing to Stacker

Thank you for your interest in contributing. Stacker is an open-source project and welcomes issues, bug fixes, new rules, and feature improvements.

---

## Getting Started

### Prerequisites

- **Node.js** 18 or higher
- **npm** 9 or higher (or pnpm/yarn)
- **Git**

### Clone and Install

```bash
git clone https://github.com/your-org/stacker.git
cd stacker
npm install
```

### Build

```bash
npm run build
```

This compiles TypeScript from `src/` and `worker/` to `dist/`.

### Run Locally

```bash
# Analyze the current directory using the local build
node dist/cli.js analyze .

# Analyze a GitHub repo
node dist/cli.js analyze https://github.com/user/repo

# Run with verbose output
node dist/cli.js analyze . --verbose
```

### Watch Mode (during development)

```bash
npm run dev
```

## Project Structure

```

├── src/                          # CLI source code (TypeScript)
│   ├── cli.ts                    # Entry point; defines all commands via Commander
│   ├── pipeline.ts               # Orchestrates the full analysis flow
│   ├── stackDetector.ts          # Detects tech stack from repo files
│   ├── analyzer.ts               # File walking, LOC counting, component/route detection
│   ├── ruleEngine.ts             # Deterministic rule checks (no AI required)
│   ├── architectureMapper.ts     # Detects patterns (serverless, JAMstack, monolith, etc.)
│   ├── reportGenerator.ts        # Formats terminal output and markdown reports
│   ├── config.ts                 # Config loading from env vars and config files
│   ├── cache.ts                  # Caches AI responses to avoid redundant calls
│   ├── rateLimiter.ts            # Client-side request rate tracking
│   ├── security.ts               # NIST 800-53 aligned security controls
│   ├── logger.ts                 # CLI logger (info, warn, error, debug)
│   ├── repoCloner.ts             # Shallow git clone for remote GitHub URLs
│   ├── auth/                     # Token storage, machine fingerprinting, request signing
│   ├── api/                      # Proxy HTTP client
│   ├── commands/                 # login and logout command implementations
│   ├── llm/                      # AI model call wrappers
│   └── prompts/                  # System prompts for AI models
│
├── worker/                       # Cloudflare Worker (proxy server)
│   ├── index.ts                  # Main worker request handler
│   ├── auth.ts                   # Token validation
│   ├── rateLimiter.ts            # Per-machine rate limiting (Durable Objects or KV)

│
├── dist/                         # Compiled output (gitignored, generated by build)
├── tsconfig.json
└── package.json
```

## How the Analysis Pipeline Works

Understanding this flow will help you know where to make changes:

1. **`cli.ts`** receives the command and calls `pipeline.ts`
2. **`repoCloner.ts`** clones the repo if a GitHub URL is provided
3. **`analyzer.ts`** walks the file tree to collect stats
4. **`stackDetector.ts`** reads `package.json`, `requirements.txt`, `go.mod`, etc. to identify the stack
5. **`architectureMapper.ts`** infers architecture patterns from the detected stack and file layout
6. **`ruleEngine.ts`** runs deterministic checks and returns findings
7. **`pipeline.ts`** sends structured metadata + rule findings to the proxy via `api/`
8. **`llm/`** models return suggestions, scores, and a summary
9. **`reportGenerator.ts`** formats and prints (or saves) the final report

## Adding a New Rule

Rules live in `src/ruleEngine.ts`. A rule is a pure function that receives context about the repo and returns a finding object or `null`.

```typescript
const myRule: Rule = ({ stack, componentCount, apiRouteCount, fileCount }) => {
  if (stack.testing === null && fileCount > 50) {
    return {
      ruleId: 'TEST_001',
      category: 'Testing',
      severity: 'warning',
      current: 'No testing framework',
      suggested: 'Vitest',
      reason: 'No test files detected in a project with significant code.',
      benefit: 'Catch regressions early. Vitest is fast and has a Jest-compatible API.',
      effort: 'medium',
      priority: 'high',
      tags: ['testing', 'quality']
    };
  }
  return null;
};
```

Add your rule to the exported `rules` array at the bottom of the file.

## Adding Stack Detection

Stack detection is in `src/stackDetector.ts`. Detection reads files like `package.json`, `requirements.txt`, `Cargo.toml`, `go.mod`, and `*.config.*` files.

To add detection for a new framework or tool, find the relevant section and add a pattern:

```typescript
// Example: detect a new CSS framework
if (deps['unocss']) {
  stack.styling = 'UnoCSS';
}
```

## PR Guidelines

- **One concern per PR.** Bug fixes, new rules, and new features should be separate PRs.
- **Describe what and why** in the PR description, not just what you changed.
- **Test your change.** Run `node dist/cli.js analyze .` against at least one real repo before submitting.
- **Follow existing code style.** The codebase uses TypeScript strict mode. Run `npm run build` to catch type errors before pushing.
- **Keep rules deterministic.** Rules in `ruleEngine.ts` must not make network calls or have side effects. They should be pure functions.
- **Do not add dependencies lightly.** Every new `node_modules` entry affects install size and supply-chain risk. Discuss in an issue first if your change requires a new package.

### Commit Message Format

Use conventional commits:

```
feat: add detection for Bun as package manager
fix: correct LOC count when symlinks are present
docs: update contributing guide with worker setup
refactor: extract fingerprint logic into separate module
```

---

## Code Style

- TypeScript strict mode is enabled (`tsconfig.json`)
- No `any` types without a comment explaining why
- Prefer `const` over `let`; avoid `var`
- Keep functions focused and small — if a function is doing more than one thing, split it
- Log user-facing messages through `logger.ts`, not `console.log` directly

## Working on the Proxy Worker

The worker lives in `worker/`. To develop and deploy the worker you need a Cloudflare account and Wrangler:

```bash
npm install -g wrangler
wrangler dev worker/index.ts   # Local development
wrangler deploy                # Deploy to Cloudflare Workers
```

Worker contributions follow the same PR process as CLI contributions.
